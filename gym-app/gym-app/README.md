# 린메스업 트래커 — 배포 가이드

이 폴더는 Claude 미리보기 안에서 만들었던 앱을 실제 웹사이트로 배포할 수 있게 정리한 프로젝트예요.

## 가장 중요한 것: 왜 이렇게 만들었나

기존 미리보기(artifact)는 코드를 수정할 때마다 저장된 데이터와의 연결이 끊길 수 있는 구조였어요.
이 프로젝트는 그 문제를 구조적으로 없앴습니다.

- 데이터는 이제 **배포된 사이트의 브라우저 저장소(localStorage)**에 저장돼요.
- 코드를 나중에 아무리 다시 배포해도, **같은 주소(도메인)로 배포하는 한 데이터는 절대 지워지지 않아요.**
- 코드와 데이터가 완전히 분리된 구조라, "기능 추가 → 데이터 삭제" 문제가 원천적으로 발생하지 않습니다.

---

## 1. 가장 쉬운 배포 방법 — Netlify Drop (계정 없이 즉시 배포)

이미 빌드까지 끝낸 `dist` 폴더가 이 프로젝트 안에 들어있어요.

1. 컴퓨터에서 https://app.netlify.com/drop 접속
2. 이 프로젝트의 `dist` 폴더를 통째로 그 페이지에 드래그 앤 드롭
3. 몇 초 안에 `https://아무이름.netlify.app` 같은 주소가 생성됨
4. 그 주소를 폰으로 열어서 **"홈 화면에 추가"** 하면 앱처럼 아이콘이 생김

이후에 계속 쓰려면 무료 계정을 만들어서 그 사이트를 "Claim"(내 소유로 등록)해두는 걸 추천해요. 등록 안 해도 당장 쓰는 데는 문제없어요.

## 2. 나중에 기능을 더 추가하고 싶을 때 — GitHub + Vercel/Netlify 연동

1. https://github.com 에서 새 저장소(repository) 생성
2. 이 프로젝트 폴더 전체(`node_modules`, `dist` 제외)를 업로드
   - GitHub 웹사이트에서 "Add file → Upload files"로 드래그해도 됨 (git 명령어 몰라도 가능)
3. https://vercel.com 또는 https://app.netlify.com 에서 "New Project" → 방금 만든 GitHub 저장소 선택
4. 빌드 명령은 자동 감지됨 (`npm run build`, 출력 폴더 `dist`) — 그대로 Deploy
5. 이후 코드를 고치고 싶으면 GitHub 저장소 파일을 수정하고 커밋하면 **자동으로 재배포**되고, 데이터는 그대로 유지됨

## 3. 로컬에서 직접 실행해보고 싶다면 (선택)

```bash
npm install
npm run dev
```

## AI 기능(음식 계산 · 외식 추천) 사용하려면

**대부분의 흔한 음식·프랜차이즈 메뉴는 API 키 없이도 무료로 즉시 계산돼요.** `src/foodDB.js`에 자주 먹는 한식·치킨·버거·카페 메뉴가 이미 들어있어서, 거기 있는 음식이면 네트워크 호출 없이 바로 결과가 나와요.

목록에 없는 음식이거나 더 폭넓은 외식 추천을 원할 때만 아래 API 키가 필요해요:

Claude.ai 미리보기 안에서는 API 키 없이 AI 호출이 됐지만, **독립된 웹사이트에서는 본인의 Anthropic API 키가 필요해요.**

1. https://console.anthropic.com/settings/keys 에서 API 키 발급 (가입 필요)
2. 배포된 사이트 → **몸 탭 → API 키 설정**에 키 붙여넣고 저장
3. 이후 음식 계산·외식 추천 정상 작동

**참고할 점**
- 이 키는 이 브라우저에만 저장되고, 사용한 만큼만 요금이 부과돼요. 개인이 하루 몇 번 쓰는 수준이면 한 달에 몇백 원~몇천 원 수준으로 매우 저렴해요.
- 키를 입력한 상태로 이 사이트 주소를 다른 사람과 공유하지 마세요. 브라우저 개발자 도구로 키가 노출될 수 있어요 (개인용 앱이라 감수 가능한 수준의 트레이드오프예요).
- 키를 넣지 않아도 다른 기능은 전부 정상 작동하고, 음식은 "직접 숫자로 입력"으로 기록할 수 있어요.

## 자주 먹는 음식을 로컬 목록에 더 추가하고 싶다면

`src/foodDB.js` 파일의 `FOOD_DB` 배열에 항목을 추가하면 돼요. Claude에게 "이 음식 로컬 목록에 추가해줘"라고 요청하면 형식에 맞춰 추가해줄 수 있어요. 자주 먹는 음식일수록 여기에 쌓아두면 API 비용이 계속 줄어들어요.

## 나중에 사이트를 업데이트하고 싶을 때 (데이터는 안전해요)

Netlify Drop으로 배포했다면, 같은 프로젝트의 "Deploys" 페이지에서 새로 빌드한 `dist` 폴더를 다시 드래그 앤 드롭하면 코드만 업데이트돼요. **저장된 기록(localStorage)은 이 배포 과정과 완전히 분리되어 있어서 전혀 영향받지 않아요.** 이게 바로 이 프로젝트로 옮긴 핵심 이유예요.

## 코드 구조 (앞으로 계속 수정할 때 참고)

```
src/
├── App.jsx      ← 모든 화면과 로직. 섹션 주석(===== 오늘 =====)으로 구분되어 있어요
│    ├── 상수/유틸 (색상 C, TYPES, 날짜 헬퍼)
│    ├── normalize() ← 데이터 구조 정의. 새 데이터 필드는 여기 먼저 추가
│    ├── App (저장/로드/자동재시도 로직)
│    ├── Today / Calendar / DayEditor / Foods(검색+Dining) / Study / Body (탭별 화면)
│    └── 공용 컴포넌트 (Card, LineChart, Bars7, FoodAI 등)
├── foodDB.js    ← 로컬 음식 DB(164개) + 매칭/검색 로직. 기본 음식 추가는 여기
│                  (앱 안에서 '음식 탭 → 내 음식 등록'으로도 추가 가능)
├── storage.js   ← localStorage 저장 어댑터 (건드릴 일 거의 없음)
└── main.jsx     ← 진입점
```

**수정할 때 규칙 하나만 기억하세요**: 새 데이터 필드를 추가할 땐 `normalize()`에 기본값을 함께 넣어야 기존 사용자의 저장 데이터와 충돌 없이 동작해요. (예: `routines: d.routines || []`)

Claude에게 이 zip을 다시 업로드하고 "여기에 ○○ 기능 추가해줘"라고 하면 이어서 개발할 수 있어요.

## 데이터 백업은 계속 챙기세요

localStorage는 안정적이지만, 브라우저 데이터를 수동으로 완전히 삭제하거나 다른 브라우저/기기로 옮기면 데이터가 안 보여요.
앱 안 **몸 탭 → 데이터 백업 → 백업 텍스트 만들기**로 가끔 백업해두는 습관은 계속 유지하는 걸 추천해요.

## 여러 기기(폰+컴퓨터)에서 같은 데이터를 보고 싶다면

지금 구조는 기기별로 데이터가 따로예요 (localStorage는 기기·브라우저 단위). 여러 기기 동기화가 필요해지면, 나중에 Supabase 같은 무료 클라우드 DB를 붙이는 다음 단계로 넘어가면 돼요. 지금 당장은 필요할 때 위 백업 기능으로 기기 간 데이터를 옮기면 됩니다.
